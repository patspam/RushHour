54=head1 DESCRIPTION
55
56Essentially, this module is a very optimized version of L<Clone::More>. By taking
57advantage of one of L<Clone::More>'s 'OPTIMIZATION_HACKS' as well as removing all
58the Pure Perl from the C<More.pm>, I was able to gain a lot of speed out of the module.
59Essentially, though, the core of the module is exactly as that of L<Clone::More>.
60
61You will see that by useing L<Benchmark::cmpthese>, I ran a simple comparison between
62L<Storable::dclone>, L<Clone::More::clone>, and L<Clone::Fast::clone>. You will (should)
63begin to see the reason why I loaded this module along side of L<Clone::More>.
64
65 Rate Storable Clone::More Clone::Fast
66 Storable 7552/s -- -39% -59%
67 Clone::More 12400/s 64% -- -33%
68 Clone::Fast 18442/s 144% 49% --
69
70For more information relative to the DESCRIPTION of this module, I recommend peeking into
71the POD written for L<Clone::More> (I took more time with it ;) )

73=head2 HISTORY
74
75As noted in L<Clone::More>, this module started as a patch for L<Clone> with repsect to a
76large memory leak a team I was working closely with at the time fell across once implemented
77the cloning into a Perl application. The unfortunate part is that I wasn't able to patch the
78L<Clone> module without a complete re-factor (I still have no idea where the leak is in Clone),
79and have not been able to get ahold of Ray Finch, the current author and supporter of L<Clone>.
80Every thing considered, I loaded up this and it's counter part L<Clone::More> - both a little
81different from one another, and both a little different from L<Clone> still.

83=head2 EXPORT
84
85=head3 clone
86
87Clone is the primary function from within the provided module. By passing a scalar reference to this
88routine, you will expect to get a returned scalar reference that will no longer have any reference to
89the originating reference. However, references deeper into the structure will still uphold the references
90within the structure.
91
92Example being:
93
94 use Clone::Fast qw( clone );
95
96 my $foo = { 'a' => 'b' };
97 my $bar = { 'a' => $foo, 'b' => $foo };
98
99 my $baz = clone( $bar );
100
101 print "\$foo and \$bar are different references\n" if ( $foo ne $bar );
102 print "\$foo->{'a'} and \$bar->{'a'} are different references\n" if ( $foo->{'a'} ne $bar->{'a'} );
103 print "\$foo->{'a'} and \$foo->{'b'} are the same, however\n" if ( $foo->{'a'} eq $bar->{'b'} );
104
105This makes sense, although this can be modified as well. By using the internal variable, BREAK_REFS, you
106are also allowed to break internal references (may break up circular references, although won't fix
107the circular reference in the originating reference).
108
109=head1 PROGRAMATIC HOOKS
110
111Much like the Perl Storable module (available in all current Perl distributions), C<Clone::Fast> allows for
112hooks that will be accessed when cloning any object that has a hook defined. This can be very handy where
113Inside Out objects would not normally be cloned. WHHAAATT???? What I mean is, only the reference of an
114object will be cloned, not the internal stash of the object. Therefore, accessors that are defined within
115an inside out object will not be cloned. There is no real safe way to do this, with the exception of cloning
116the entire class stash, breaking more things than it will fix. Again, the reference of the object will be
117fully cloned, and the object it's self will be a new reference, although it will be an empty object. Subsiquently,
118such as most inside out objects, the blessed reference is of a scalar type; an integer indicating the object id.
119When cloning this, you would end up with two objects of the same type with the same object id. The hooks
120have been added in an attempt to prevent this from happening.
121
122=head2 CLONEFAST_clone
123
124Again, much like L<Storable> (though a little better, I hope), the function will be called *AFTER* the clone
125operation has completed on the object being cloned. The routine will have two scalar references passed
126via the stack, representing both the cloned object as well as the source of the clone. This *should* allow
127for the programatic manipulation of the object before it gets returned to the caller, or placed into the
128refering structure.
129
130As an example, I will use the following object to define a 'hooked' object:
131
132 package Hookable;
133
134 use strict;
135 use warnings;
136
137 use Clone::Fast qw( clone );
138
139 sub new { bless {}, shift }
140
141 sub CLONEFAST_clone {
142
143 # Where clone is the cloned object from the source, where source
144 # was the originating reference
145 my ( $clone, $source ) = @_;
146
147 # I am going to pretend the source has a list of defined methods,
148 # of which I want to clone and transfer to the clone; outside
149 # of the blessed hash-refrence that is the source of the object
150 $clone->$_( clone( $source->$_() ) ) for ( qw( get_method_1 get_method_2 get_method_3 ) );
151
152 # At this point, the cloned object will also have a set of cloned
153 # fields from the source. If, by chance, any of the values of the
154 # defined attribtes are other 'Hookable' objects, the same routine
155 # will be called on that object as well.
156
157 # The API requires me to return the new $clone, this will be returned to
158 # the caller
159 return $clone;
160 }
161
162Using the package from above, I will now use an example of a script where I will demonstrate how
163the whole thing comes together:
164
165 #!/usr/bin/perl -w
166
167 use strict;
168
169 use Clone::Fast qw( clone );
170
171 my $hookable = Hookable->new();
172 $hookable->{'hash_stuff'} = 'some value';
173
174 my $structure = {
175 'hookable' => $hookable,
176 'new' => Hookable->new(),
177 'deeply' => {
178 'hookable' => $hookable,
179 'new' => Hookable->new();
180 },
181 };
182
183 my $cloned = clone( $sturcture );
184
185This script will demonstrate a number of things. 1.) C<Clone::Fast::clone> will, automagically call the hook
186on all instances of the Hookable. Though the hash_stuff key will automatically be cloned before the hook is ever
187called. Subsiquently, the hashes in both values of hookable in the hash will be references of one another, though
188not references to the originating object. The Hookable->new() object, on the other hand, will not be referenced
189to anything of the similar like.
190
191As a secondary note, it was originally thought to allow for hooks to show up before and after the cloning of the
192object. Though, that would allow for the full change of the cloning type; this would be very bad. Also, given
193that it is somewhat reasonable to believe hooks will only be used with inside out objects, we can also assume the
194cloning of a simple referent will be so lightweight that there will still be the benifit of having clone hook into
195the object. If anyone has beef with this paradigm, let me know and I'll change it.

197=head1 CONFIGURATION VARIABLES
198
199=over
200
201=item $Clone::Fast::ALLOW_HOOKS
202
203The C<ALLOW_HOOKS> variable will allow for the toggling behavior, telling C<Clone::Fast> to check for
204hooks when cloning objects. (See C<PRGRAMATIC HOOKS> for more details). The varialble will default
205to 'on', where C<Clone::Fast> will always check each object for hooks defined within the object.
206
207 use Clone::Fast qw( clone );
208
209 $Clone::Fast::ALLOW_HOOKS = 1; # No need, this is default
210
211 my $object = HasHooks->new();
212
213 package HasHooks;
214
215 use strict;
216 use warnings;
217
218 sub new { bless {}, shift }
219
220 sub CLONEFAST_clone {
221 my ( $clone, $source ) = @_;
222
223 # Re-assigning the reference will now return the reference from the
224 # C<Clone::Fast::clone> when cloning a HasHooks object, rather than
225 # a cloned reference to the object.
226 $clone = { 'object' => $clone };
227 return $clone;
228 }
229
230=item $Clone::Fast::BREAK_REFS
231
232 use Clone::Fast qw( clone );
233 $Clone::Fast::BREAK_REFS = 1;
234
235 my $foo = { 'a' => 'b' };
236 my $bar = { 'a' => $foo, 'b' => $foo };
237
238 my $baz = clone( $bar );
239
240 print "\$foo and \$bar are different references\n" if ( $foo ne $bar );
241 print "\$foo->{'a'} and \$bar->{'a'} are different references\n" if ( $foo->{'a'} ne $bar->{'a'} );
242 print "\$foo->{'a'} and \$foo->{'b'} are no longer the same\n" if ( $foo->{'a'} ne $bar->{'b'} );
243
244You will see that by adding the BREAK_REFS flag, you will change the overall behavior of the routine.
245The BREAK_REFS flag must, simply, have truthfullness (as far as Perl is concerned) in order to be 'on'.
246
247Whereas:
248
249 $Clone::Fast::BREAK_REFS = 1;
250
251 # Will do the same thing as:
252
253 $Clone::Fast::BREAK_REFS = 'yes';
254
255 # Will do the same thing as:
256
257 $Clone::Fast::BREAK_REFS = ( 2 != 1 );
258
259Albeit handy, this feature may also slow down the module by some degree. Therefore, there is some flexibility
260into whether or not you need to use this, and the functionality can be compiled out of the object; speeding up
261the cloning ability. Therefore, Re-compiling the mdule without MINDFUL_REFS will increase the speed of the
262module by a degree of 3x! If you KNOW you will never use the $Clone::Fast::BREAK_REFS and are confident
263with manually installing Perl modules from source, it is recommended you do so. There are comments in the XS source
264that will detail how to do this.
265
266This configuration only applies to the C<Clone::Fast::clone> routine.

301=head1 GOTCHAS/WARNINGS
302
303=over
304
305=item bless()'ed references (Perl objects)
306
307This module works great for blessed references, how ever the paradigm changes when trying to clone
308inside out objects (or Conway's 'flywaight' style of object creation). Clone does not, nor will not,
309clone the stash of an object's class; this would break more than anything. Given this, HOOKS have
310been provided in order to programatically handle wierd stuff like this. I am hoping applications,
311developers and all of the like whom are using inside out objects will know what the heck it is I'm
312talking about here. There is a lot more information about this in the PROGRAMATIC HOOKS section.
313
